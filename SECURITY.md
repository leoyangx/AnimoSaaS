## 报告安全漏洞

我们非常重视安全问题。如果您发现了安全漏洞，请遵循以下步骤：

### 1. 不要公开披露

**请勿**通过以下方式公开披露安全漏洞：
- GitHub Issues
- 公开论坛
- 社交媒体
- 公开的聊天室

### 2. 私密报告

请通过以下方式报告安全漏洞：

**邮件：** security@example.com

**报告内容应包括：**
- 漏洞类型（如 SQL 注入、XSS、CSRF 等）
- 受影响的组件或文件
- 漏洞的详细描述
- 复现步骤
- 概念验证代码（如果可能）
- 潜在影响评估
- 建议的修复方案（可选）

### 3. 响应时间

- **确认收到**：24 小时内
- **初步评估**：3 个工作日内
- **修复计划**：7 个工作日内
- **补丁发布**：根据严重程度，14-30 天内

### 4. 披露政策

我们遵循**协调披露**原则：

1. 您报告漏洞
2. 我们确认并评估
3. 我们开发和测试修复
4. 我们发布安全补丁
5. 我们公开披露（在修复发布后 30 天）
6. 您可以公开披露（在我们公开披露后）

## 安全最佳实践

### 对于用户

1. **使用强密码**
   - 至少 12 个字符
   - 包含大小写字母、数字和特殊字符
   - 不要重复使用密码

2. **保护环境变量**
   - 不要将 `.env` 文件提交到版本控制
   - 使用强随机的 `JWT_SECRET`
   - 定期轮换密钥

3. **保持更新**
   - 及时更新到最新版本
   - 关注安全公告
   - 订阅安全通知

4. **生产环境配置**
   - 设置 `DISABLE_SECURE_COOKIE=false`
   - 启用 HTTPS
   - 配置防火墙
   - 限制数据库访问

### 对于开发者

1. **输入验证**
   - 使用 Zod 验证所有输入
   - 不信任任何用户输入
   - 验证文件上传

2. **认证和授权**
   - 使用中间件保护路由
   - 检查用户权限
   - 使用 CSRF 保护

3. **数据库安全**
   - 使用参数化查询（Prisma）
   - 不要在日志中记录敏感信息
   - 加密敏感数据

4. **依赖管理**
   - 定期运行 `npm audit`
   - 及时更新依赖
   - 审查新依赖

## 已知安全特性

### 已实施的安全措施

✅ **认证和授权**
- JWT 令牌认证
- 基于角色的访问控制（RBAC）
- HTTP-only cookies
- 中间件路由保护

✅ **输入验证**
- Zod 运行时验证
- 文件类型和大小限制
- 文件名清理

✅ **速率限制**
- 登录端点：15分钟5次
- 管理端点：1分钟100次
- 下载端点：1分钟10次

✅ **密码安全**
- bcrypt 哈希（12 rounds）
- 密码复杂度要求
- 安全的邀请码生成

✅ **CSRF 保护**
- CSRF token 验证
- SameSite cookie 属性

✅ **数据库安全**
- Prisma ORM（防止 SQL 注入）
- 软删除支持
- 审计日志
